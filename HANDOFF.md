## 1. What This Is

A mobile-first progressive-web fleet builder for **Battlefleet Gothic Remastered** — a specialist Games Workshop tabletop game. The user selects a faction, picks ships, manages points, and validates their list against composition rules. Like Battlescribe but purpose-built for BFG with a much better UX.

**Reference product:** An existing Dropfleet Commander Fleet Builder at `D:\wargaming\Web Apps\Dropfleet-Builder\` — match its feel and capability. **Do NOT touch or modify that project — ever.**

## 4. UX Principles (Non-Negotiable)

### Non-blocking validation
Users add ships in any order and fix violations later. Nothing is blocked. Warnings appear inline as amber badges on the card AND in the validation panel. They never prevent an action.

### Validation hidden on empty fleet
The validation panel is hidden until at least one ship/commander is added. No nagging on empty lists.

### Split pts bar
The pts progress bar has two segments: **brass** (valid pts) and **red** (invalid pts from rule-violating ships). Purely informational. When over the fleet limit, the entire tracker turns red (pts-used text, remaining text, and bar).

### Greek letter suffixes
When the same ship class appears multiple times, each instance gets a Greek suffix at render time: "Lunar Class Cruiser α", "Lunar Class Cruiser β". The suffix is **not stored** in fleet data — computed at render from position in list.

```js
const GREEK = ['α','β','γ','δ','ε','ζ','η','θ','ι','κ','λ','μ','ν','ξ','ο','π'];
```

### Picker tab memory
Last active category tab is remembered between picker opens. FAB does **not** reset the tab.

### Better ship subtitle
Ship cards show `"8 Hits · 2 Shields · 6+Front/ 5+"` not category name. Category is already obvious from the section header.

### Over-limit red state
When total pts exceed fleet limit: pts-used turns red, remaining text turns red, valid bar segment turns dark red.

### Sticky section headers
Category headers (BATTLESHIPS, CRUISERS, ESCORTS) use `position: sticky; top: 0` so you always see which section you're in.

### Smooth expand/collapse
Ship detail panels use `max-height: 0 → 2000px` + `opacity 0 → 1` CSS transition. ~150ms. Never `display` toggle.

### Desktop: always-visible picker
On desktop (≥900px) the picker is always visible in the right panel. No FAB, no screen switching. Mobile: FAB opens picker as full-screen overlay; back closes it.

### Fleet options ···
Topbar ··· opens dropdown: Rename (`window.prompt` — replace with inline edit), Duplicate (deep-clone + " (Copy)" suffix), Delete (`window.confirm`).

### Add Anyway pattern
In the picker, if a ship would violate a rule (e.g. battleship without enough cruisers), the add button turns amber and reads "Add to Fleet Anyway" with a warning box above explaining the issue. Ship is still addable.

## 5. Fleet Data Model

Stored as JSON array in `localStorage['bfg-fleets']`. Each fleet:

```json
{
  "id": 1720000000000,
  "name": "Gothic Patrol",
  "faction": "Imperial Navy",
  "fleetList": "Gothic Sector Fleet",
  "limit": 1500,
  "commander": null,
  "ships": [
    {
      "shipId": "ship-id-string",
      "qty": 1,
      "upgrades": [
        { "id": "upgrade-id", "name": "Prow Torpedoes", "pts": 5 }
      ],
      "_idx": 1720000000001
    }
  ],
  "squadrons": [
    {
      "shipId": "ship-id-string",
      "count": 3,
      "upgrades": []
    }
  ],
  "created": "2024-07-09T12:00:00.000Z"
}
```

**Key decisions:**
- Escorts → `squadrons[]`, not `ships[]`. Each unique escort type gets one entry; `count` tracks how many (2–6).
- Fleet Commander → single `commander` object, not in `ships[]`. Only one per fleet.
- `_idx` is `Date.now()` tiebreaker for card identity (allows duplicate ship types as separate cards).
- `qty` field exists but UI only does qty=1 currently — no +/- controls for non-escort ships yet.

**Commander object:**
```json
{
  "shipId": "ship-id-string",
  "name": "Admiral Ravensburg",
  "pts": 50,
  "rerolls": 2
}
```

## 8. Validation Rules

`validateFleet(fleet)` returns `[{ type: 'err'|'warn', msg, affectsCategory? }]`. All non-blocking.

| Rule | Type | Trigger | Message |
|---|---|---|---|
| Fleet Commander | `err` | total pts > 750 AND `fleet.commander === null` | "Fleet Commander required for fleets above 750 pts" |
| Battleship ratio | `err` | battleships > 0 AND cruisers < battleships × 3 | "N Battleship(s) require M cruisers — add X more" · `affectsCategory: 'Battleship'` |
| Escort min | `warn` | any squadron `count < 2` | "[Ship] squadron needs at least 2 ships (currently N)" |
| Escort max | `warn` | any squadron `count > 6` | "[Ship] squadron exceeds maximum of 6 ships (N)" |

**Cruiser types that count toward battleship ratio:**  
`Cruiser`, `Battlecruiser`, `Light Cruiser`, `Heavy Cruiser`, `Grand Cruiser`

**pts split:** When battleship ratio is violated, all battleships in `fleet.ships` are flagged invalid. Their pts appear as the red bar segment. `ptsSplit(fleet)` → `{ valid, invalid }`.

## 14. BSData Parsing Notes

Parser: `scripts/parse-bsdata.js`. Two-pass approach:

**Pass 1 — GST (`battlefleetgothic.gst`)**  
Builds lookup table of all ship entries. All entries are `hidden="true"` — it's a shared library, not directly selectable.

**Pass 2 — `.cat` files (one per faction)**  
Ships appear as `<entryLink>` elements referencing GST entries. Parser resolves each link against the lookup, merging pts/categories/upgrades.

**Critical: `parseCost()` must use direct children only:**
```js
function parseCost(entryEl) {
  for (const costsEl of childrenNamed(entryEl, 'costs')) {
    for (const costEl of childrenNamed(costsEl, 'cost')) {
      if (attr(costEl, 'name') === 'pts') {
        const v = parseFloat(attr(costEl, 'value'));
        return isNaN(v) ? 0 : v;
      }
    }
  }
  return 0;
}
```
Using `descendantsNamed` instead breaks Lunar Class (and others) — it finds nested 0-pt upgrade costs before the real 180-pt ship cost.

**Do NOT filter `hidden="true"` in `.cat` files** — Orks, Tyranids, etc. have ships marked hidden at top level (revealed via fleet-list modifiers). Filtering them out gives 0 ships for those factions.

**Run parser:** `node scripts/parse-bsdata.js` from project root. Outputs to `data/ship_database.json`.
